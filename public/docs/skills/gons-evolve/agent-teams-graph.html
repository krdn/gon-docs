서브에이전트 선택 시

  • 결과만 필요한 독립적인 집중 작업
  • 에이전트 간 협업·토론이 불필요
  • 토큰 비용을 절약해야 할 때
  • 순차 작업 또는 의존성이 많은 작업
  • 동일 파일을 여러 에이전트가 편집하는 경우

에이전트 팀 선택 시

  • 병렬 탐색이 진정한 가치를 더할 때
  • 팀원들이 결과를 공유하고 서로 검증해야 할 때
  • 독립적으로 진행 가능한 모듈 구현
  • 경쟁 가설로 디버깅 — 다수 이론 병렬 검증
  • 프론트엔드, 백엔드, 테스트 크로스레이어 작업
에이전트 팀 워크플로우
팀이 시작되고 작업을 완료하기까지의 전체 흐름을 시각화합니다.

팀 실행 흐름

1

요청 또는 제안

사용자가 팀 생성 요청하거나 Claude가 병렬 작업이 유용하다고 판단해 제안

2

사용자 승인 필수

Claude는 사용자 승인 없이 팀을 생성하지 않음

3

팀 생성 및 팀원 스폰

리드가 공유 태스크 목록 생성 후 팀원 인스턴스 스폰

4

태스크 분배 및 병렬 실행

팀원들이 독립 태스크를 클레임하고 병렬로 작업

5

결과 종합

리드가 결과 수집·종합. 팀원에게 shutdown 요청

6

팀 정리 (Clean up)

리드가 공유 리소스 제거. 활성 팀원이 있으면 실패하므로 먼저 종료 필수

디스플레이 모드

In-process (기본)

메인 터미널 내 실행. Shift+Down으로 팀원 전환

요구사항: 없음 (모든 터미널)

Split panes

각 팀원이 별도 창. 동시 출력 확인 가능

요구사항: tmux 또는 iTerm2
설정 가이드
에이전트 팀을 활성화하고 첫 번째 팀을 시작하는 단계별 가이드입니다.

기능 활성화

에이전트 팀은 기본 비활성화입니다. settings.json에 환경 변수를 추가하세요.

~/.claude/settings.json필수
{ "env": { "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1" } }

디스플레이 모드 설정 (선택)

기본값은 "auto"입니다. tmux 세션 내에서 실행 중이면 split panes, 아니면 in-process 자동 선택.

~/.claude/settings.json선택
{ "teammateMode": "in-process" // "in-process" | "tmux" | "auto" } # 또는 플래그로 일회성 지정 claude --teammate-mode in-process

split panes — tmux 설치 (선택)

split-pane 모드를 사용하려면 tmux 또는 iTerm2가 필요합니다.

tmux 설치
# Ubuntu/Debian sudo apt install tmux # macOS brew install tmux
iTerm2: it2 CLI 설치 후 Settings → General → Magic → Enable Python API 활성화

첫 번째 팀 시작

자연어로 팀 구성을 요청합니다. 팀원 수와 역할을 명시하거나 Claude에게 위임할 수 있습니다.

예시 프롬프트
CLI 도구 설계를 다양한 관점에서 탐색해줘. UX 전문가, 기술 아키텍트, 악마의 변호인 역할로 에이전트 팀 3명을 만들어서 분석해줘.

팀 정리 (Clean up)

작업 완료 후 반드시 리드를 통해 팀을 정리합니다. 활성 팀원이 있으면 먼저 종료해야 합니다.

팀 정리 순서
# 1단계: 팀원 종료 Ask all teammates to shut down. # 2단계: 팀 리소스 정리 Clean up the team.
주의: 팀원이 cleanup을 직접 실행하면 팀 컨텍스트가 잘못 해석될 수 있습니다. 항상 리드를 통해 정리하세요.
사용 사례 예시
병렬 탐색이 실질적인 가치를 더하는 대표적인 시나리오들입니다.
🔍

병렬 코드 리뷰

단일 리뷰어는 한 번에 한 유형의 이슈에 집중하는 경향이 있습니다. 도메인을 나누면 보안, 성능, 테스트 커버리지가 동시에 철저히 검토됩니다.
🔒 보안 검토자
⚡ 성능 검토자
🧪 테스트 검토자

Create an agent team to review PR #142. Spawn three reviewers:
- One focused on security implications
- One checking performance impact
- One validating test coverage
Have them each review and report findings.

독립 모듈 병렬 개발

새 기능이 여러 독립 모듈로 구성될 때, 각 팀원이 다른 파일 집합을 소유하면 충돌 없이 병렬 구현이 가능합니다.
🖥 프론트엔드
⚙️ 백엔드
🧪 테스트
파일 충돌 방지: 두 팀원이 같은 파일을 편집하면 덮어쓰기가 발생합니다. 각 팀원이 서로 다른 파일 집합을 소유하도록 태스크를 분배하세요.

다각도 연구 및 리뷰

문제의 다양한 측면을 여러 팀원이 동시에 조사하고, 결과를 공유하며 서로 도전합니다.
💡 UX 전문가
🏗 기술 아키텍트
😈 악마의 변호인
프롬프트 예시
I'm designing a CLI tool that helps developers track TODO comments across
their codebase. Create an agent team to explore this from different angles: one
teammate on UX, one on technical architecture, one playing devil's advocate.
베스트 프랙티스
에이전트 팀을 효과적으로 사용하기 위한 검증된 가이드라인입니다.
1

팀원에게 충분한 컨텍스트 제공

팀원은 리드의 대화 이력을 상속받지 않습니다. CLAUDE.md, MCP 서버, 스킬은 자동 로드되지만, 태스크별 세부 사항은 스폰 프롬프트에 직접 포함해야 합니다. 파일 경로, 기술 스택, 집중 영역을 명시적으로 전달하세요.

2

적절한 팀 크기 선택 (3-5명 권장)

팀원 수에 비례하여 토큰 비용이 증가합니다. 너무 많으면 조율 오버헤드가 커집니다. 대부분의 워크플로우에서 3-5명이 최적이며, 팀원당 5-6개 태스크가 적절합니다.

15개 독립 태스크라면 3명으로 시작하세요. 5-6 tasks per teammate keeps everyone productive without excessive context switching.
3

태스크 크기를 적절하게 조정

너무 작으면 조율 오버헤드가 이득을 초과합니다. 너무 크면 팀원이 체크인 없이 너무 오래 작업해 낭비 위험이 커집니다. 명확한 결과물(함수, 테스트 파일, 리뷰)을 가진 자족적인 단위가 이상적입니다.

4

팀원 완료를 기다리기

리드가 팀원을 기다리지 않고 스스로 태스크를 구현하기 시작할 수 있습니다. 이럴 때 직접 지시하세요: "Wait for your teammates to complete their tasks before proceeding"

5

연구/리뷰로 시작하기

처음이라면 코드 작성이 없는 명확한 경계의 작업부터 시작하세요. PR 리뷰, 라이브러리 연구, 버그 조사는 병렬 탐색의 가치를 보여주면서 동시 구현의 조율 문제를 피할 수 있습니다.

6

파일 충돌 방지

두 팀원이 같은 파일을 편집하면 덮어쓰기가 발생합니다. 각 팀원이 다른 파일 집합을 소유하도록 태스크를 분배하세요. 크로스레이어(프론트/백엔드/테스트) 작업은 자연스럽게 충돌을 피합니다.

7

지속적으로 모니터링 및 조정

팀을 방치하면 낭비된 노력이 커집니다. 팀원 진행 상황을 확인하고, 잘못된 접근 방식을 수정하며, 결과가 도착하는 대로 종합하세요. 계획 승인 기능으로 리스크가 높은 작업은 구현 전 검토하세요.

제한 사항 및 트러블슈팅
현재 알려진 제한 사항과 문제 해결 방법을 확인하세요. 에이전트 팀은 실험적 기능입니다.

알려진 제한 사항

세션 재개 불가 (in-process)

/resume과 /rewind는 in-process 팀원을 복원하지 않습니다. 재개 후 리드가 존재하지 않는 팀원에게 메시지를 보내려 할 수 있습니다.

태스크 상태 지연

팀원이 가끔 태스크를 완료로 표시하지 못해 의존 태스크가 블록될 수 있습니다. 수동으로 상태를 업데이트하거나 리드에게 nudge 요청하세요.

느린 종료

팀원은 현재 요청이나 도구 호출을 완료한 후 종료합니다. 시간이 걸릴 수 있습니다.

세션당 하나의 팀만 가능

리드는 한 번에 하나의 팀만 관리할 수 있습니다. 새 팀 시작 전 현재 팀을 정리하세요.

중첩 팀 불가

팀원은 자체 팀이나 팀원을 스폰할 수 없습니다. 팀 관리는 리드만 가능합니다.

리드 고정

팀을 생성한 세션이 그 팀의 리드입니다. 팀원을 리드로 승격하거나 리더십을 이전할 수 없습니다.

권한은 스폰 시 설정

모든 팀원은 리드의 권한 모드로 시작됩니다. 스폰 시점에 팀원별 권한 설정 불가. 스폰 후에는 개별 변경 가능.

Split panes 터미널 제한

VS Code 통합 터미널, Windows Terminal, Ghostty에서 split-pane 모드가 지원되지 않습니다.

트러블슈팅

팀원이 나타나지 않을 때

  • in-process: Shift+Down으로 이미 실행 중인 팀원 확인
  • 태스크가 팀이 필요할 만큼 복잡한지 확인
  • split panes 요청 시 tmux 설치 확인: which tmux
  • iTerm2: it2 CLI 설치 및 Python API 활성화 확인

권한 프롬프트 너무 많을 때

팀원 스폰 전에 권한 설정에서 공통 작업을 미리 승인하여 마찰을 줄이세요.

에러로 팀원이 정지할 때

  • in-process: Shift+Down으로 해당 팀원 출력 확인
  • split: 해당 창 클릭 후 직접 확인
  • 추가 지시 전달 또는 대체 팀원 스폰

리드가 작업 완료 전 종료할 때

리드가 태스크가 실제로 완료되기 전에 팀이 끝났다고 판단할 수 있습니다. 계속 진행하도록 지시하거나, 팀원 완료를 기다리도록 명시적으로 요청하세요.

tmux 세션이 남아있을 때

팀 종료 후 tmux 세션이 남아있으면 수동으로 정리:

tmux ls tmux kill-session -t <session-name>
