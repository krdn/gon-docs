→ Context Flow
서브에이전트가 생성되고 작업을 수행하고 결과를 반환하는 전체 흐름입니다.
1

사용자 요청 수신

사용자가 메인 대화에서 작업을 요청합니다. Claude는 등록된 서브에이전트들의 description 필드를 확인하여 매칭되는 에이전트를 찾습니다.

User: "코드 리뷰 해줘" ↓ description 매칭 Match: code-reviewer "Reviews code for quality"
2

독립 컨텍스트 생성

새로운 컨텍스트 윈도우가 생성됩니다. 서브에이전트는 자신의 시스템 프롬프트(Markdown body)만 받습니다. Claude Code의 전체 시스템 프롬프트는 전달되지 않습니다.

Context Window (New) ├─ System: SKILL.md body ├─ Tools: Read, Grep, Glob ├─ Model: sonnet └─ Env: working dir, OS
3

독립 실행

서브에이전트가 허용된 도구만 사용하여 작업을 수행합니다. 메인 대화의 컨텍스트를 오염시키지 않습니다. 서브에이전트는 다른 서브에이전트를 스폰할 수 없습니다.

Sub-agent working... → git diff (Bash) → Read modified files → Grep for patterns → Analyze & summarize ✗ Cannot spawn agents
4

결과 반환

서브에이전트가 완료되면 요약 결과만 메인 대화에 반환됩니다. 상세 도구 호출 결과는 서브에이전트의 독립 트랜스크립트에만 저장됩니다. Agent ID가 반환되어 나중에 재개(resume)할 수 있습니다.

Main Context ← Summary "3 critical issues found" "2 warnings, 1 suggestion" Agent ID: agent-abc123 → Resume 가능
5

트랜스크립트 저장

서브에이전트의 전체 대화 이력은 ~/.claude/projects/{project}/{session}/subagents/agent-{id}.jsonl에 저장됩니다. 메인 대화 압축과 독립적이며, 기본 30일 후 자동 정리됩니다.

Transcript File ~/.claude/projects/ {project}/{session}/ subagents/ agent-abc123.jsonl ← 독립 저장, 30일 유지
📂 Scope & Priority
동일한 이름의 서브에이전트가 여러 위치에 있으면, 우선순위가 높은 쪽이 승리합니다.

📄 Frontmatter Anatomy
서브에이전트 정의 파일의 구조와 각 필드의 역할을 시각적으로 분석합니다.
code-reviewer.md
--- name: code-reviewer description: Reviews code for quality tools: Read, Grep, Glob, Bash disallowedTools: Write, Edit model: sonnet permissionMode: default maxTurns: 20 memory: user background: false isolation: worktree skills: - api-conventions hooks: PreToolUse: - matcher: "Bash" hooks: - type: command command: "./validate.sh" --- # System Prompt (Markdown body) You are a senior code reviewer...
*

name (필수)

고유 식별자. 소문자 + 하이픈만 사용. 동일 이름 시 우선순위 높은 쪽이 승리.

*

description (필수)

Claude가 위임 시점을 판단하는 핵심. "use proactively" 포함 시 자동 위임 강화.

T

tools / disallowedTools

허용 목록(allowlist) 또는 거부 목록(denylist). 생략 시 모든 도구 상속.

M

model

haiku(빠름/저렴), sonnet(균형), opus(최고성능), inherit(부모 모델). 기본: inherit.

P

permissionMode

default, acceptEdits, dontAsk, bypassPermissions, plan. 부모가 bypass면 오버라이드 불가.

⚙

memory / skills / hooks / isolation

영속 메모리(user/project/local), 스킬 프리로드, 라이프사이클 훅, worktree 격리.

✍

Markdown Body (시스템 프롬프트)

서브에이전트가 받는 유일한 프롬프트. Claude Code 전체 시스템 프롬프트는 전달 안 됨.

🔧 Tools Control
서브에이전트의 도구 접근을 허용 목록, 거부 목록, Agent 스폰 제한으로 세밀하게 제어합니다.

✓ Allowlist (tools)

tools: Read, Grep, Glob, Bash # 이 도구만 사용 가능 # 생략 시 모든 도구 상속
명시된 도구만 허용합니다. 생략하면 부모의 모든 도구를 상속합니다.

✗ Denylist (disallowedTools)

disallowedTools: Write, Edit # 이 도구를 제외 # tools에서 또는 상속에서 제거
허용 목록이나 상속된 도구에서 특정 도구를 명시적으로 제거합니다.

👥 Agent 스폰 제한

# 특정 에이전트만 스폰 가능 tools: Agent(worker, researcher) # 모든 에이전트 스폰 가능 tools: Agent # Agent 생략 → 스폰 불가
--agent로 실행하는 메인 에이전트에만 적용. 서브에이전트는 다른 서브에이전트 스폰 불가.

🚫 특정 에이전트 비활성화

// settings.json { "permissions": { "deny": [ "Agent(Explore)", "Agent(my-custom)" ] } }
permissions.deny로 빌트인/커스텀 에이전트를 비활성화합니다.

🔄 Lifecycle & Hooks
서브에이전트의 라이프사이클 이벤트와 훅을 통한 세밀한 제어 흐름입니다.

▶ Start Phase

SubagentStart settings.json에서 정의. 메인 세션에서 실행. matcher로 특정 에이전트 타겟팅.
skills 로드 frontmatter의 skills 필드 → 전체 스킬 내용을 컨텍스트에 주입.
memory 로드 memory 필드 → MEMORY.md 첫 200줄을 컨텍스트에 포함.
▼

⚙ Execution Phase (반복)

PreToolUse 도구 실행 전. matcher로 특정 도구 타겟. exit 2 → 실행 차단.
Tool 실행 허용된 도구만 실행. 제한된 도구 호출 시 거부.
PostToolUse 도구 실행 후. 린터 실행, 로깅 등에 활용.
▼ maxTurns 도달 또는 작업 완료

📦 Auto-compaction (필요 시)

~95% 용량 자동 압축 트리거. CLAUDE_AUTOCOMPACT_PCT_OVERRIDE로 조기 설정 가능.
▼

■ Stop Phase

Stop (fm) frontmatter의 Stop 훅 → 런타임에서 SubagentStop으로 변환.
SubagentStop settings.json에서 정의. 메인 세션에서 실행. 정리 작업(cleanup)에 활용.
결과 반환 요약 + Agent ID를 메인 대화에 반환. 트랜스크립트는 .jsonl로 저장.
🧠 Persistent Memory
서브에이전트가 대화를 넘어서 지식을 축적하고, 시간이 지날수록 더 똑똑해지는 메커니즘입니다.
🌎

user

~/.claude/agent-memory/<name>/
모든 프로젝트에서 기억을 유지합니다. 프로젝트를 넘나들며 축적된 패턴과 인사이트를 활용할 수 있습니다.
⭐ 권장 기본값
📁

project

.claude/agent-memory/<name>/
프로젝트에 특화된 지식을 저장합니다. 버전 관리에 포함되어 팀원과 공유할 수 있습니다.
팀 공유 가능
🔒

local

.claude/agent-memory-local/<name>/
프로젝트 특화 + 개인 전용. 버전 관리에서 제외되므로 민감한 메모나 로컬 설정에 적합합니다.
Private

⚙ Memory 활성화 시 자동 동작

1 시스템 프롬프트에 주입
메모리 읽기/쓰기 지시가 자동으로 추가됩니다.
2 MEMORY.md 자동 로드
첫 200줄이 컨텍스트에 포함됩니다. 200줄 초과 시 큐레이션 지시.
3 도구 자동 활성화
Read, Write, Edit 도구가 자동으로 활성화되어 메모리를 관리합니다.
4 지식 축적
패턴, 아키텍처 결정, 디버깅 인사이트를 대화를 넘어 보존합니다.
▶▮ Foreground vs Background
서브에이전트는 포그라운드(차단) 또는 백그라운드(동시) 모드로 실행할 수 있습니다.

▶ Foreground (기본)

  • ●메인 대화가 차단됨 (완료까지 대기)
  • ●권한 프롬프트가 실시간으로 사용자에게 전달
  • ●AskUserQuestion으로 질문 가능
  • ●결과를 기다려야 하는 작업에 적합
  • ●기본 동작 — 별도 설정 불필요

▮ Background

  • ●메인 대화 계속 진행 가능
  • ●시작 전 필요한 권한을 사전 승인
  • ●미승인 권한은 자동 거부
  • ●AskUserQuestion 실패 (하지만 계속 진행)
  • ●"백그라운드로 실행" 또는 Ctrl+B
  • ●background: true로 항상 백그라운드

💡 백그라운드 실패 시 복구

백그라운드 서브에이전트가 권한 부족으로 실패하면, resume으로 포그라운드에서 재시도할 수 있습니다.

"Continue that review" → Agent ID를 기반으로 이전 컨텍스트 전부 유지한 채 포그라운드에서 재개.

비활성화: CLAUDE_CODE_DISABLE_BACKGROUND_TASKS=1
💥 Usage Patterns
서브에이전트를 효과적으로 활용하는 3가지 핵심 패턴과 실전 사례입니다.
📦

Pattern 1: 대량 출력 격리

컨텍스트 절약

테스트 실행, 로그 분석, 문서 가져오기 같은 대량 출력을 서브에이전트에 격리합니다. 상세 결과는 서브에이전트 컨텍스트에만 남고, 요약만 반환됩니다.

User: "테스트 실행하고 실패한 것만 보고해" Claude: → test-runner 서브에이전트에 위임 Result: "3개 실패: auth.test.ts:42, ..." (전체 테스트 로그는 서브에이전트에만 보존)
🚀

Pattern 2: 병렬 리서치

속도 향상

독립적인 조사를 여러 서브에이전트가 동시에 수행합니다. 각각 독립 탐색 후 Claude가 결과를 종합합니다.

User: "인증, DB, API 모듈을 병렬로 분석해" Claude: → 3개 서브에이전트 동시 스폰 [auth-explorer] → 인증 모듈 분석 [db-explorer] → DB 모듈 분석 [api-explorer] → API 모듈 분석 Claude: → 3개 결과 종합

⚠ 많은 서브에이전트 결과가 모두 반환되면 메인 컨텍스트 소비 주의. 지속적 병렬은 Agent Teams가 적합.

🔗

Pattern 3: 체이닝

순차 파이프라인

여러 서브에이전트를 순차 실행하여 파이프라인을 구성합니다. 첫 번째 결과를 Claude가 다음 서브에이전트에 전달합니다.

User: "성능 문제 찾고 수정해줘" Step 1: code-reviewer → 성능 이슈 탐지 "3개 N+1 쿼리 발견" Step 2: optimizer → 이슈 수정 (컨텍스트 전달) "3개 쿼리 배치 처리로 변환 완료"

서브에이전트 중첩 불가 → 체이닝은 항상 메인 대화에서 조율.

🔄

Pattern 4: Resume (재개)

연속성

완료된 서브에이전트를 이전 컨텍스트 전체를 유지한 채 재개합니다. 새 인스턴스를 시작하지 않고 이어서 작업합니다.

User: "인증 모듈 리뷰해줘" Agent: → 리뷰 완료 (ID: abc123) User: "그 리뷰 이어서 권한 로직도 분석해" Claude: → agent-abc123 resume 이전 도구 호출, 결과, 추론 모두 보존

⚠ Constraints & Limits
서브에이전트를 사용할 때 반드시 알아야 하는 핵심 제약사항입니다.
🚫

중첩 스폰 불가

서브에이전트는 다른 서브에이전트를 스폰할 수 없습니다. 중첩이 필요하면 메인 대화에서 체이닝으로 해결하세요.

🔌

스킬 비상속

부모 대화의 스킬을 자동으로 상속하지 않습니다. frontmatter의 skills 필드에 명시적으로 나열해야 합니다.

📄

제한된 시스템 프롬프트

Claude Code의 전체 시스템 프롬프트를 받지 않습니다. 오직 자신의 Markdown body + 기본 환경 정보만 받습니다.

🔒

bypassPermissions 불변

부모가 bypassPermissions를 설정하면 자식 서브에이전트가 이를 오버라이드할 수 없습니다. 항상 부모 설정이 우선.

🚫

백그라운드 권한 제한

백그라운드 서브에이전트는 사전 승인된 권한만 사용. 미승인 권한은 자동 거부되고, AskUserQuestion도 실패합니다.

⏰

시작 지연 (Cold Start)

서브에이전트는 새로운 컨텍스트에서 시작하므로 코드베이스 파악에 시간이 필요합니다. 지연이 중요하면 메인 대화가 적합.

⭐ Best Practices

1 하나의 작업에 집중

각 서브에이전트는 한 가지 일을 잘 하도록 설계. 범용보다 전문화가 효과적.

2 Description을 상세하게

Claude가 위임 시점을 정확히 판단하는 핵심. "use proactively" 포함 권장.

3 도구 접근 최소화

필요한 도구만 부여. 읽기 전용 에이전트에 Write/Edit을 줄 이유가 없음.

4 버전 관리에 체크인

프로젝트 서브에이전트를 .claude/agents/에 저장하고 팀과 공유.
