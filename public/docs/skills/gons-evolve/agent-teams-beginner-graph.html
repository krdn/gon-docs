여러 AI를 팀으로 구성해 복잡한 작업을 동시에 처리하는 방법 — 초보자도 이해하기 쉬운 완전 입문 가이드

공식 문서 기반 · 2025년 최신 버전 · Claude Code 전용
에이전트 팀이란 무엇인가?
개념을 쉬운 비유로 먼저 이해하고, 실제 동작 방식을 살펴봅니다
"

핵심 비유: 회사 팀 vs 혼자 일하는 직원

혼자 일하는 Claude는 만능 직원 한 명과 같습니다. 모든 걸 혼자 처리하니 순서대로 할 수밖에 없어요.

에이전트 팀은 여러 명의 전문가로 구성된 팀입니다. 팀장(Team Lead)이 지시하면, 각 팀원(Teammate)이 각자 맡은 부분을 동시에 처리합니다. 보안 전문가, 성능 전문가, 테스트 전문가가 한꺼번에 코드 리뷰를 하는 것처럼요.

4가지 핵심 개념
👑

팀 리드 (Team Lead)

처음 대화를 시작하는 Claude 세션입니다. 팀을 만들고, 팀원에게 업무를 배정하고, 결과를 종합합니다.

비유: 프로젝트 매니저. 모든 팀원의 진행 상황을 파악하고 조율합니다.
🤖

팀원 (Teammate)

리드가 생성한 독립적인 Claude 인스턴스(분신)입니다. 각자의 독립적인 컨텍스트 창에서 작업하며, 서로 직접 메시지를 주고받을 수 있습니다.

비유: 각 분야 전문가. 보안, 성능, 테스트 담당자가 각자 병렬로 작업합니다.
📋

공유 작업 목록 (Task List)

모든 팀원이 공유하는 할 일 목록입니다. 팀원들은 여기서 작업을 가져와 완료 처리합니다. 동시 수정을 방지하는 파일 잠금 기능이 내장되어 있습니다.

비유: 사무실 화이트보드. "할 일", "진행 중", "완료" 항목이 모두에게 보입니다.
📬

메일박스 (Mailbox)

팀원 간 메시지를 주고받는 시스템입니다. 특정 팀원에게 1:1로 메시지를 보내거나, 전체 브로드캐스트할 수 있습니다.

비유: 사내 메신저. 팀원이 메시지를 보내면 자동으로 전달됩니다. 수동 확인 불필요.
에이전트 팀이 빛나는 순간
🔍

리서치 & 리뷰

여러 관점의 동시 탐색

비유: 법정에서 여러 검사가 각자 다른 증거를 동시에 조사하는 것처럼
  • PR 리뷰: 보안, 성능, 테스트 커버리지를 동시에 검토
  • 버그 조사: 여러 가설을 동시에 검증
  • 경쟁 분석: 여러 라이브러리를 동시에 조사
🏗️

독립적 모듈 개발

겹치지 않는 파일 병렬 구현

비유: 건설 현장에서 전기, 배관, 목공 팀이 각자 구역을 동시에 작업하는 것처럼
  • 프론트엔드 + 백엔드 + 테스트를 동시에 구현
  • 서로 다른 마이크로서비스 개발
  • 독립적인 기능 모듈 병렬 작성
🐛

경쟁 가설 디버깅

다양한 원인을 동시에 추적

비유: 의사들이 희귀병 진단을 위해 각자 다른 가설로 검사를 동시에 진행하는 것처럼
  • 각 팀원이 서로 다른 버그 원인 가설을 검증
  • 팀원들이 서로의 이론을 반박하며 최종 원인 수렴
  • 편향 없는 독립적 조사로 정확도 향상
🔗

크로스 레이어 조율

프론트 · 백엔드 · 테스트 동시 작업

비유: 레스토랑에서 주방, 홀 서빙, 계산대가 동시에 운영되는 것처럼
  • 프론트엔드 UI 변경 + 백엔드 API 수정을 동시에
  • 데이터베이스 스키마 변경과 마이그레이션 병렬 처리
  • 여러 레이어에 걸친 대규모 리팩토링
⚠️

에이전트 팀이 비효율적인 경우: 순서대로 처리해야 하는 작업, 같은 파일을 수정하는 작업, 의존성이 많은 작업에서는 단일 세션이나 서브에이전트가 더 효과적입니다. 팀을 만들면 조율 오버헤드와 더 많은 토큰 비용이 발생하기 때문입니다.

에이전트 팀 vs 서브에이전트
상황에 따라 어떤 방식을 선택해야 할지 명확하게 비교합니다
💡

핵심 질문: 작업자들이 서로 대화할 필요가 있나요? 없다면 서브에이전트가 더 저렴하고 효율적입니다. 팀원들이 서로 정보를 공유하고 이론을 반박해야 한다면 에이전트 팀을 사용하세요.

🤖 서브에이전트 (Sub-Agents)
단일 세션 내 헬퍼
작동 방식
메인 에이전트가 서브에이전트를 생성 → 작업 수행 → 결과를 메인에게만 보고
팀원 간 소통
불가 — 모든 소통은 메인 에이전트를 거쳐야 함
비용
낮음 — 결과만 메인 컨텍스트에 반환
적합한 경우
결과만 필요한 집중적 작업, 빠른 조회/검증, 단순 반복 작업
비유
상사가 직원에게 "이 파일 정리해줘" 라고 지시하는 것. 직원들끼리는 말 안 함.
VS
👥 에이전트 팀 (Agent Teams)
독립적 협업 팀
작동 방식
팀 리드가 팀을 구성 → 팀원들이 공유 할 일 목록에서 작업을 가져와 완료
팀원 간 소통
직접 가능 — 팀원이 서로 메시지를 보내고 협력
비용
높음 — 각 팀원이 독립적인 컨텍스트 창 보유
적합한 경우
토론과 협력이 필요한 복잡한 작업, 경쟁 가설 검증, 다방면 리뷰
비유
여러 전문가가 회의실에서 각자 의견을 내고 토론하는 것. 직접 대화 가능.
항목 서브에이전트 에이전트 팀
컨텍스트 창 독립 각자의 컨텍스트, 결과만 반환 완전 독립 각자의 전체 세션
팀원 간 소통 메인 에이전트를 통해서만 팀원끼리 직접 메시지 가능
조율 방식 메인이 모든 작업을 관리 공유 작업 목록으로 자체 조율
토큰 비용 낮음 높음 (팀원 수에 비례)
최적 활용 결과만 중요한 집중 작업 토론·협력이 필요한 복잡한 작업
사용자 직접 접근 리드를 통해서만 각 팀원에게 직접 메시지 가능
설정 복잡도 낮음 중간 (활성화 필요)

선택 가이드 — 나에게 맞는 방식은?

Q1. 작업자들이 서로 발견한 내용을 공유하거나 반박해야 하나요? 예 → 에이전트 팀
Q2. 단순히 "이 작업 해줘" 라고 지시하고 결과만 받으면 충분한가요? 예 → 서브에이전트
Q3. 토큰 비용이 가장 중요한 제약 조건인가요? 예 → 서브에이전트
Q4. 독립적으로 각자 진행할 수 있는 큰 작업이 3개 이상 있나요? 예 → 에이전트 팀
단계별 시작 가이드
처음부터 따라하면 에이전트 팀을 사용할 수 있습니다
1

기능 활성화 (기본값: 비활성)

에이전트 팀은 실험적 기능으로, 기본적으로 꺼져 있습니다. 설정 파일(settings.json)에 환경 변수를 추가해서 켜야 합니다. settings.json 위치: 일반적으로 ~/.claude/settings.json

settings.json — 에이전트 팀 활성화
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
ℹ️

실험적 기능이란? 아직 완성 단계가 아니라는 뜻입니다. 세션 재개나 팀 종료 시 일부 제한이 있을 수 있으니, 중요한 프로젝트에서는 주의해서 사용하세요.

2

화면 표시 모드 선택

팀원들의 작업 화면을 어떻게 볼지 결정합니다.

In-Process 모드 (기본)

모든 팀원이 같은 터미널 안에서 동작합니다. Shift+Down으로 팀원 간 전환합니다. 어떤 터미널에서도 동작합니다.

설치 필요 없음

Split Panes 모드

각 팀원이 별도의 창(pane)에 표시됩니다. 모든 팀원의 작업을 동시에 볼 수 있습니다. tmux 또는 iTerm2가 필요합니다.

tmux 설치 필요
settings.json — 모드 설정 (선택사항)
{
  "teammateMode": "in-process" // 또는 "tmux", "auto"
}

// 단일 세션에서만 적용하려면:
claude --teammate-mode in-process
3

첫 번째 팀 만들기

Claude에게 자연어로 팀 구성을 요청합니다. 구체적인 역할과 작업을 명시할수록 효과적입니다. Claude가 팀 구성, 작업 배정, 조율을 자동으로 처리합니다.

초보자 추천 예시 — 코드 리뷰 팀
// 이 텍스트를 Claude에게 입력하세요
PR #142를 검토하는 에이전트 팀을 만들어줘.
팀원 3명:
- 보안 취약점 전담 검토자
- 성능 영향 전담 검토자
- 테스트 커버리지 전담 검토자
각자 검토 후 결과를 공유해줘.
💡

초보자 팁: 처음에는 코드를 작성하는 팀이 아니라 코드를 검토하는 팀으로 시작하세요. 경계가 명확하고, 파일 충돌 위험이 없으며, 병렬 탐색의 가치를 바로 체감할 수 있습니다.

4

팀원과 직접 소통하기

팀이 구성되면, 각 팀원에게 직접 추가 지시를 보내거나 진행 상황을 확인할 수 있습니다.

In-Process Shift+Down — 팀원 간 전환 | Enter — 팀원 세션 보기 | Ctrl+T — 작업 목록 보기
Split Pane 팀원 창을 클릭하면 해당 팀원에게 직접 입력 가능
계획 승인 요구 (중요한 작업에 권장)
// 팀원이 실제 코드를 변경하기 전에 계획을 먼저 승인받게 합니다
인증 모듈을 리팩토링하는 아키텍트 팀원을 생성해줘.
변경 전에 반드시 계획 승인을 받아야 해.

// 팀원이 계획을 수립 → 리드에게 승인 요청 → 리드가 검토 후 승인/거절
5

팀 종료 및 정리

작업이 완료되면 반드시 팀을 정리해야 합니다. 항상 리드(메인 세션)를 통해서 정리하세요. 팀원이 직접 정리하면 리소스 불일치가 발생할 수 있습니다.

팀 종료 순서
// 1단계: 각 팀원에게 종료 요청

// 2단계: 모든 팀원 종료 확인 후 팀 정리
팀을 정리해줘

// tmux 세션이 남아있다면 수동으로 정리:
tmux ls // 세션 목록 확인

팀 크기는 3-5명이 최적

팀원이 많을수록 토큰 비용이 선형으로 증가하고 조율 복잡도가 높아집니다. 대부분의 작업에서 3-5명이 가장 효율적입니다.

1-2
소형
명확한 분업이 되는 작업, 간단한 리뷰
3-5
권장 크기
대부분의 작업에 최적. 병렬성과 조율 비용의 균형
6+
주의 필요
진짜 필요할 때만. 비용과 조율 오버헤드 급증
황금 비율: 팀원 1명당 5-6개 작업이 적절합니다. 15개 독립 작업이라면 팀원 3명 구성을 권장합니다.
📝

팀원에게 충분한 컨텍스트를 제공

팀원은 리드의 대화 내역을 모릅니다. 스폰 시 필요한 모든 정보를 포함해야 합니다.

파일 충돌 방지 — 각 팀원이 다른 파일 담당

두 팀원이 같은 파일을 동시에 수정하면 한 팀원의 작업이 덮어씌워집니다. 처음부터 파일 담당 구역을 명확히 나누세요.

위험한 설계
팀원 A: api/users.js 수정
팀원 B: 동일한 api/users.js 수정
→ 덮어쓰기 발생!
안전한 설계
팀원 A: frontend/ 담당
팀원 B: backend/ 담당
팀원 C: tests/ 담당
👀

팀을 방치하지 말고 지속 모니터링

자동화가 많이 되어 있지만, 팀을 너무 오래 방치하면 낭비가 발생할 수 있습니다. 진행 상황을 주기적으로 확인하고, 방향이 어긋난 팀원을 조기에 수정하세요.

🔔

팀원이 리드보다 먼저 구현을 시작하면: "팀원들이 작업을 완료할 때까지 기다려줘" 라고 리드에게 지시하세요.

문제 해결 가이드
자주 발생하는 문제와 해결 방법, 그리고 알려진 제한 사항
자주 묻는 질문

초보자 공통  팀원이 화면에 나타나지 않아요

▼
1. In-process 모드에서는 이미 실행 중인 팀원이 있을 수 있습니다. Shift+Down을 눌러 팀원을 전환해보세요.
2. Claude에게 준 작업이 팀을 만들 만큼 복잡한지 확인하세요. Claude가 단독으로 처리 가능하다 판단하면 팀을 만들지 않을 수 있습니다.
3. Split-pane 모드를 요청했다면 tmux 설치 여부를 확인하세요: which tmux
4. iTerm2를 사용한다면 it2 CLI 설치와 Python API 활성화를 확인하세요.

주의  허가(Permission) 프롬프트가 너무 많이 뜹니다

▼

팀원의 허가 요청이 리드로 계속 올라오면 작업 흐름이 끊어집니다.

1. 팀원을 스폰하기 전에 settings.json의 권한 설정에서 자주 사용하는 도구를 미리 허가합니다.
2. 허가한 도구 목록을 확인하고, 팀원이 필요로 하는 도구가 포함되어 있는지 점검합니다.

자주 발생  팀원이 오류 후 멈춰 있습니다

▼

팀원이 오류를 만난 후 복구하지 못하고 멈출 수 있습니다.

1. In-process: Shift+Down으로 해당 팀원 세션으로 이동해서 출력을 확인합니다.
2. Split-pane: 해당 팀원 창을 클릭해서 오류 내용을 확인합니다.
3. 해당 팀원에게 직접 추가 지시를 보내거나, 작업을 이어받을 새 팀원을 스폰합니다.

자주 발생  리드가 팀원이 끝나기 전에 직접 구현을 시작합니다

▼

리드가 팀원을 기다리지 않고 직접 작업을 시작하는 경우가 있습니다.

1. 리드에게 직접 지시합니다: "팀원들이 모든 작업을 완료할 때까지 기다려줘"
2. 초기 프롬프트에 "팀원 작업 완료 후 진행" 이라는 조건을 명시합니다.

tmux  팀 종료 후 tmux 세션이 남아 있습니다

▼

정리가 완전히 되지 않아 tmux 세션이 남을 수 있습니다.

1. 남아있는 세션 목록 확인: tmux ls
2. 해당 세션 강제 종료: tmux kill-session -t <세션이름>

중요  세션을 재개했더니 팀원이 없어졌습니다

▼

이것은 알려진 제한 사항입니다. In-process 팀원은 /resume이나 /rewind로 복원되지 않습니다.

1. 리드가 존재하지 않는 팀원에게 메시지를 보내려 할 수 있습니다. 새 팀원을 스폰하라고 지시하세요.
2. 중요한 작업 중에는 세션 재개보다 현재 세션을 유지하는 것을 권장합니다.
알려진 제한 사항 (실험적 기능)
🚫

세션 재개 불가

/resume, /rewind로 in-process 팀원은 복원되지 않습니다. 세션 재개 후 리드가 없는 팀원에게 메시지를 보내려 할 수 있습니다.

⏳

작업 상태 지연

팀원이 완료된 작업을 "완료"로 표시하지 않아 다음 작업이 블록될 수 있습니다. 수동으로 작업 상태를 업데이트하거나 리드를 통해 팀원을 독려하세요.

🐢

종료가 느릴 수 있음

팀원은 현재 요청이나 도구 호출이 끝난 후에 종료합니다. 즉시 종료되지 않을 수 있으니 기다려주세요.

1️⃣

세션당 팀 1개

리드는 한 번에 하나의 팀만 관리할 수 있습니다. 새 팀을 만들기 전에 현재 팀을 정리해야 합니다.

🚷

중첩 팀 불가

팀원은 자신의 팀을 만들 수 없습니다. 팀 생성은 리드만 할 수 있습니다.

🖥️

Split Pane 터미널 제한

VS Code 통합 터미널, Windows Terminal, Ghostty에서는 Split Pane 모드가 지원되지 않습니다. In-Process 모드를 사용하세요.

🎯

처음 시작하는 분들을 위한 한 줄 조언

코드 리뷰 팀으로 시작하세요. 파일 충돌 없이, 병렬 탐색의 가치를 바로 체감할 수 있습니다. PR 하나를 보안·성능·테스트 전문가 3명이 동시에 검토하는 것을 직접 경험해보면, 에이전트 팀이 왜 강력한지 즉시 이해할 수 있습니다.
